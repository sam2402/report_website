Testing Strategy

Testing Strategy 

To ensure that Volvo's carbon emission dashboard meets the desired standards outlined in our MoSCoW lists and client requirements, it is imperative to conduct comprehensive tests on the client-side application in terms of its usability, functionality, and stability. 

 

Fully Automated Tests: 

  • Unit Test 

  • Integration Test 

  • End-to-end Test 

 

Semi-Automated Tests: 

  • Performance Test 

  • Responsive Design Test 

  • Browser Compatibility Test 

  • User Acceptance Test 

 

Unit Test 

As mentioned in the implementation section, our backend fetches the data from azure via different routes. Thanks to the well-established flask framework, we were able to extract the data returned by the routes and test them.   

  

In this test we use the Pytest package as our testing framework.  


In this case, we can ensure that the backend could fetch the data properly from every route.  


Integration Test 

To ensure that the frontend and backend would work properly once they were merged, we introduced Cypress as our integration test tool 

 

By checking the frontend was able to access all the resource groups provided from Azure, we can confirm that every carbon data stream required by different charts in the dashboard was supported.  

 

End-to-end Test 

To ensure that the program responds properly when user using the dashboard, end-to-end testing was introduced into our project. 


We simulate user interaction by testing each interactable element and checking the response from dashboard to make sure every interactable element works properly.  


Performance Test 

Although fetching data from Azure or generating suggestions using GPT3 can take a significant amount of time, leading to potential delays in displaying the data, we still conducted the performance tests, to evaluate the reliability of our frontend programs. 


Thanks to React Developer Tools, we can easily check the time taken by each component to render. On the welcome page, the main task is to render the carbon emissions map. We conducted a render speed test for this page, and the results are displayed in the following graphs. 


It is important to note that the yellow parts indicate longer rendering times. However, the tool colors components based on relative render times, so a yellow component may not necessarily have a long absolute render time. Here is another rendering analysis for the same page. 


The two graphs show that rendering the map for different countries takes the most time. Despite this, the page still renders quite quickly, taking only around 50ms, as indicated on the right side of the graphs. 


In addition, we tested the speed to render the prediction page. 


As shown in these screenshots, the two line charts take the longest to render on this page, as indicated by the two longest vertical bars at the top. Despite this, they were still being rendered quickly, taking only around 40ms each. 


For the page with no chart, the resource group page renders more quickly than the previous two pages, as expected. It takes around 30ms only to render whole page. The most time-consuming rendering component on this page, the frame of the multiple select bar, takes only 0.8ms to render.  


In summary, we were pleased to find that all the different parts of our dashboard rendered quickly, demonstrating the excellent frontend design of our application.

Conclusion 

Both our test users and client claimed that the dashboard was so well finished, and accepted the case of not being able to display emission data quickly due to the use of external databases. They believe that with the help of this dashboard, tracking real-life carbon emissions will be much easier and clearer.
