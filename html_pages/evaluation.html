Critical evaluation of the project

 

 

User interface and experience


Because one of our MoSCoW lists is that the dashboard application should be easy to for non-technical Volvo's employees, we made sure throughout the design process that our dashboard was designed with the customer experience in mind.


So, our entire dashboard is designed to be simple and clear, easy to navigate to different pages, and the navigation bar can be hidden for better display of the charts. Only setup environment & launch dashboard is a bit complicated for the non-technical users. Our UI design does a great job of hiding cumbersome information from the customer. 

For more details, please checkout User Acceptance Testing.

 

Functionality


We have focused on key requirements during the design process to create a dashboard that would showcase past carbon emissions.


At the same time, we also met the majority of the client's needs within the MoSCoW list, with different graphs to show past and present carbon emissions data, forecast future carbon emissions data and g However, because the dataset is so small, we can only use the weaker However, because the dataset is so small, we can only use the weaker ML model for ours. Also, the predicted future data can only be predicted for 2-3 days, otherwise the accuracy of the data cannot be guaranteed. Then again, also due to the lack of data, it was not possible to complete the calculation of the amount of carbon emissions reduction predicted to be approximately equal to the number of flights.

 

Stability & Efficiency


Regarding these two parts, they are also two aspects that we paid a lot of attention to during the development process. First, our project is a Dashboard, and usually, this type of project should perform well in terms of stability and efficiency. When we are processing past data, it is fast. Then we didn't use our own database for data storage but Azure provided by the customer, so Volvo could store the data on Azure in the future as it does now to ensure some stability. Then we set the program to automatically update the cache of resources, resource groups and predictions every hour to ensure long-term reliability, efficiency and speed. Our only program with a long response time is probably the ML model that calculates future carbon emissions, but the time series analysis model we chose is the SARIMA model, which has a better overall performance for system load, CPU consumption, and processing speed, and it only takes us about 30 seconds per run to predict the data of one resource (because the data set we have access to is also too small), so we set the program to run once every hour to predict future data, a behavior that also ensures a long-term reliability and efficiency. In addition, our 100% passed end-to-end tests for all possible user interaction gives us strong confidence in the stability of our product as well. But if our data sources could be larger and more accurate, we would feel that the stability would be even better.

 

Compatibility

Regarding the Compatibility of our project, our product is a website type dashboard, which is well done to work in Firefox/ Chrome/ Safari, but we don't have a Windows/IOS/Android version. Back to the project itself the functions and libraries designed should all show good backward versioning and cross-platform compatibility. 

 

Maintainability

In our project we use react for the frontend. In React, we embrace a component-based architecture to structure our application. Each component is built individually, ensuring modularity and independence. Once developed, these components are connected via APIs. By implementing thorough unit testing for each component, we guarantee that they can function autonomously. This approach allows us to swiftly identify and resolve any bugs encountered, without impacting the performance of other components in the system.

 

Project management

We did a good job of project management, assigning different parts (front-end, back-end, ML model) to different members, and we made good use of Github's branch management feature to ensure there were no conflicts between them. And the regular intra-group meetings kept things well together, as well as the initial Gantt chart gave us a clear goal during the project.

Future work

If we could be given more time, we would consider the following improvements to this project.


Build our own database:

The reason is because the existing Azure only allows us to have access to the past month's carbon emissions data, which is slightly weak if we only have the past month's carbon emissions data to use as a dataset to analyse and predict the future carbon emissions data. If we can create a database to extract the data from each month and place it in the database, this can greatly expand our dataset, which will make our forecast data more accurate and better capture the seasonal data. This change would greatly improve the accuracy of our forecast data.

Multiple file types export function: 

In some cases, customers may want to export past carbon emission data graphs and future projected carbon emission data graphs and compare them, if we are given longer time, we add a file export function such as JSON file format to simply show the data, or PDF format to show the graph comparison, or DOC file to show the past carbon emission data. emissions data trends over time.

Localisation:

Because Volvo is a global company, this dashboard will also be used globally, if we can give more time, we can develop a multi-language version, such as French, Spanish, Chinese, Japanese and so on. This will facilitate the use of our dashboard in the company.

Mobile version: 

We are now a web-based dashboard, if we can give more time to complete a version that can be used on mobile phones/iPad, it will allow our customers to observe the past data in a more portable way.

Own ML API for generating suggestions: 

We are now specifying rules and generating prompts to pass to OpenAI's API to ask for suggestions to reduce carbon emissions. If we have more time and more data, we can try to build our own proposal generation model, which will make the generated proposals more relevant to our project.
